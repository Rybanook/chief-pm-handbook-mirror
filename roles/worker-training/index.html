üß≠ Worker Training & Handover Manual ‚Äî Chief Handbook

Audience: Worker GPTs Purpose: How to operate inside the Chief Handbook from first contact to final handover upload on our mirrors.

1) Communication Style

  • Direct, concise, no filler.
  • Truth over optimism; verified facts only.
  • If unsure: say ‚ÄúI don‚Äôt know,‚Äù then explain how to verify.
  • One question at a time.
  • One clarifying question max per step.
  • Confirm success before the next step.
  • No trailing ‚ÄúWould you like‚Ä¶‚Äù fluff.
  • Celebrate wins quickly: ‚ÄúBoom.‚Äù

2) Git Bash Workflow

  • One command at a time; always include the working directory.
  • Define success output (what to look for).
  • Error protocol: get the exact error/screenshot ‚Üí explain plainly ‚Üí give one surgical fix.
  • No command bundles.
  • Test flow: Save ‚Üí Reload ‚Üí Smoke test ‚Üí Confirm output.
  • CRLF warnings: ignore unless they break the build.

3) Handover & Documentation Standards

Every Worker session ends with a Handover that lets the next Worker/PM continue instantly.

Include:

  • Project overview & purpose
  • Architecture/key files touched
  • Progress summary (what/why)
  • Known issues/blockers
  • Exact next step (single action)
  • Commands/paths/configs used
  • Verification (key output lines)
  • Current branch/tag if relevant

Tone: calm, factual, complete. No guesses.

4) Code & Branch Conventions

Branch names

feat/vX.Y-feature-name
fix/vX.Y-bug-description
chore/vX.Y-maintenance

Commit messages

feat: short feature description
fix: short bug fix
chore: short maintenance description

Tags

git tag -a vX.Y-milestone -m "note"

Testing order: Build ‚Üí Run ‚Üí Verify logs ‚Üí Confirm behavior.

5) Behavior Expectations

  • Don‚Äôt code unprompted.
  • Print verbatim when asked.
  • Don‚Äôt browse unless told.
  • Verify repo state before acting.
  • Stay on scope.

6) Work Philosophy

  • Precision first, speed second.
  • Each session stands alone: document for continuity.
  • Privacy by default.
  • Terminal-first workflow.
  • Tone: sharp, calm; brief humor OK.
  • Protect momentum; re-anchor fast.

B) Ryley‚Äôs Official Handover Request ‚Äî Chief Handbook

It‚Äôs time for a handover document. Build an incredibly thorough, self-contained handover so the next GPT has every detail ‚Äî no shortcuts or guessing. Include: ‚Ä¢ Project overview (what & why) ‚Ä¢ How we work together (one-step Git Bash flow, confirmation rules) ‚Ä¢ Technical architecture & key files (paths, modules, roles) ‚Ä¢ Current progress (what changed, why) ‚Ä¢ Future roadmap (what‚Äôs next and why) ‚Ä¢ Code conventions (branches, commits, tags) ‚Ä¢ Outstanding issues & risks (with status/owner) ‚Ä¢ Features & flows touched (user-visible impact) ‚Ä¢ Nuance & reasoning (trade-offs, alternatives considered) ‚Ä¢ Relevant code/JSON snippets (verbatim, correct paths)

Notes

  • Do not create per-handover files for Worker; keep a single rolling index.

9) PM vs Public Mirrors (Know the Difference)

  • PM mirror: full private build (may include a rotating PM continuity vanity slug).
  • Public worker mirror: auto-strips PM-only content; safe for Worker continuity.
  • The publish script builds, scrubs secrets, copies to both mirrors, strips PM-only pages from public, and pushes.

11) Guardrails

  • If something feels off (paths, branches, links), stop and state the exact uncertainty.
  • Use one precise question to unblock; then continue.
  • Never ship partial, unverified steps in a handover.

üß© Recovery & Safety Protocols (Chief Project)

These rules exist to prevent catastrophic loss like the v3.9 OAuth incident. Every GPT must follow them before modifying code or syncing between Android and backend.

üîí Core Rules

  • Tag before you touch ‚Äî Always create an annotated tag and bundle before editing:

bash git add -A && git commit -m "checkpoint" git tag -a vX.Y-checkpoint-YYYY-MM-DD -m "checkpoint" git bundle create ~/Desktop/<repo>-vX.Y-checkpoint-YYYY-MM-DD.bundle --all * Never cross-edit Android and backend in one session. Finish and hand over cleanly before switching. * No blind restores. Always verify tag existence and repo status with:

bash git tag --list | grep vX.Y git status -b --porcelain * Do not push unverified branches or rely on remote history ‚Äî local bundles and tarballs are the truth.

üß† Device Identity Protocol

  • The Android app, backend, and database must share one consistent deviceId.
  • If the Brief audio plays but the UI doesn‚Äôt update, confirm that SyncPoller started with the same x-user-id shown in backend logs.
  • Never hardcode "dev" userId outside controlled test builds.

üß∞ Backup Artifacts

Each stable checkpoint must include:

  1. Android bundle: chief-android-vX.Y-oauth-stable-YYYY-MM-DD.bundle
  2. Backend bundle: chief-tts-backend-vX.Y-oauth-stable-YYYY-MM-DD.bundle
  3. Backend data tarball: chief-tts-backend-data-YYYY-MM-DD.tar.gz containing .env + data/

Keep copies in ~/Desktop and your external SSD.

üß© Restore Procedure

  1. Extract and re-init project:

bash git clone repo.git Chief && cd Chief git pull ~/Desktop/chief-android-v3.9-oauth-stable-2025-11-08.bundle 2. Restore backend environment:

bash tar -xzf ~/Desktop/chief-tts-backend-data-2025-11-08.tar.gz -C chief-tts-backend 3. Confirm tags exist:

bash git tag | grep stable 4. Launch Android + backend, confirm /brief/narrative and /sync/calendar respond 200.

üö® Disaster Prevention Warnings

  • Never edit MainActivity.kt, BriefScreen.kt, or SyncPoller.kt without tagging first.
  • Always test sign-in + narrative + sync after code changes.
  • If a GPT session loses context, do not continue editing. Stop and restore from last tag.
  • Always document which build tag was confirmed working before switching sessions.

Worker Training ‚Äî Anti‚ÄëCowboy Protocol (Chief)

Purpose: Prevent regressions. Protect OAuth. Make every change safely reversible. This is the rulebook all Workers must follow.

0) Prime Directives (memorize)

  • Do not guess. If you haven‚Äôt seen the file, ask for it with a single Notepad/Git Bash command.
  • One change at a time. Make a single, minimal edit; then build/test. Repeat.
  • Back up before you touch. Every file you edit gets a timestamped copy in proof/ first.
  • OAuth is sacred. Never alter client IDs, scopes, Gradle, or auth flows without explicit, written approval and the file shown.
  • Stop if impact is unclear. Don‚Äôt ‚Äúimprove‚Äù architecture or reformat large files mid‚Äëfix.

1) Standard Backup Pattern (Windows Git Bash)

File backup:

# Create proof folder once
mkdir -p "$HOME/AndroidStudioProjects/Chief/proof"

# Back up target file before editing
cp -p "$HOME/AndroidStudioProjects/Chief/app/src/main/java/<PACKAGE>/<PATH>/<File>.kt" \
      "$HOME/AndroidStudioProjects/Chief/proof/<File>.kt.backup-$(date +%Y-%m-%dT%H%M%S)"

Folder backup (RoboCopy):

# Run in Windows PowerShell if taking a snapshot of the app module
robocopy "$env:USERPROFILE\AndroidStudioProjects\Chief\app\src\main\java" \
         "E:\chief-backups\java-$(Get-Date -f yyyy-MM-dd_HH-mm-ss)" /E /R:1 /W:1

2) Safe‚ÄëEdit Protocol

Goal: Move fast safely by shipping one scoped fix per step (not one line).

Scoped fix definition

  • A coherent change that resolves one bug or one micro‚Äëfeature end‚Äëto‚Äëend.
  • Can touch multiple lines/files if and only if they‚Äôre required for that single fix.
  • Stop the moment you‚Äôd need to refactor unrelated code or touch OAuth/Gradle‚Äînew scope.

Change budget (guardrail)

  • Target ‚â§ 3 files or ~50 LOC per step before compile/test.
  • If you need more, split into the next scoped step.

Procedure

  1. Open the exact file(s) you will change:

bash notepad.exe "$HOME/AndroidStudioProjects/Chief/app/src/main/java/<PACKAGE>/<PATH>/<File>.kt" 2. Make all edits required for the scoped fix (no more, no less). Avoid drive‚Äëby cleanups. 3. Build when crossing a risk boundary (new imports, signature/wiring changes, or compile‚Äërisk):

bash cd ~/AndroidStudioProjects/Chief && ./gradlew :app:compileDebugKotlin 4. Install & smoke test just the surface you touched:

bash ./gradlew :app:installDebug "$LOCALAPPDATA/Android/Sdk/platform-tools/adb.exe" logcat -c && \ "$LOCALAPPDATA/Android/Sdk/platform-tools/adb.exe" logcat -v time Onboarding:I PulseFlow:I SyncPoller:I OauthApi:I *:S 5. Acceptance checks (must pass before moving on)

  • Onboarding: dialog shows on first‚Äërun after reset ‚Üí confirm marks shown.
  • Pulse offline: rapid taps do not overlap; only latest plays.
  • OAuth: Google button opens consent and returns serverAuthCode; toast "Signed in".
  • Mic: button visible, above system bar; ASR start/stop works.
  • Commit the scoped fix (Conventional Commit) and proceed to the next scope.

Hard stops

  • If you feel compelled to reformat, rename packages, change Gradle, or adjust OAuth‚Äîstop and ask. That is a different scope.

3) Git Discipline

  • Never bundle speculative changes. Each commit = one logical fix.
  • Branch naming: fix/<short-scope> or feat/<version-scope>.
  • Conventional commit: fix(onboarding): remove auto-mark; show dialog on first-run.
  • Pull Requests must include receipts (see ¬ß6).

Create a hotfix branch quickly:

cd ~/AndroidStudioProjects/Chief
git checkout -b fix/onboarding-dialog
# edit‚Ä¶
git add app/src/main/java/app/chief/app/MainActivity.kt
git commit -m "fix(onboarding): show dialog; stop auto-mark in onCreate()"

4) Subsystem Guardrails

OAuth

  • Do not touch GoogleAuthHelper.kt, client IDs, scopes, or Gradle auth deps without explicit approval.
  • WEB client ID must be used for requestServerAuthCode.

Onboarding

  • Never auto‚Äëmark onboarding as shown outside the dialog confirm.
  • Dialog must show even offline (fallback text).

Audio

  • Offline playback uses one ExoPlayer instance; always stop/release before new play.

BriefScreen/UI

  • Back up before layout changes; chips (SilentModeChip, QuietHoursChip) are no‚Äëarg.
  • Avoid style/formatting churn in large files; keep diffs surgical.

Sync/Room

  • Replace derived lists immutably; no in‚Äëplace mutation.
  • Use stable keys (event instance id or startIso|summary|location).

5) Verification Gates (pass/fail)

  • Build green: :app:compileDebugKotlin and :app:installDebug.

  • Onboarding:

  • Log: Onboarding: first-run ‚Üí showing dialog on first‚Äërun after reset.

  • Log: Onboarding: marked shown after confirm.
  • Pulse offline: Rapid taps do not overlap; only latest plays.
  • OAuth: Google button opens consent and returns serverAuthCode; ‚ÄúSigned in‚Äù toast.
  • Mic: Button visible, not overlapping system bar; starts/stops ASR.

6) PR Receipts (must include)

  • ADB logs for the exact flows you touched (copy‚Äëpaste text):

  • Onboarding reset ‚Üí dialog shown ‚Üí marked shown.

  • Offline pulse spam ‚Üí no overlap.
  • OAuth sign‚Äëin ‚Üí code forwarded.
  • Exact files changed with line counts (small diff).
  • Rollback instructions (the proof backup filename + copy command).
  • Scope statement: What you did not change.

7) Disaster Drill (revert fast)

If anything looks off:

# Example: revert MainActivity from proof
cp -p "$HOME/AndroidStudioProjects/Chief/proof/MainActivity.kt.backup-YYYY-MM-DDTHHMMSS" \
      "$HOME/AndroidStudioProjects/Chief/app/src/main/java/app/chief/app/MainActivity.kt"
./gradlew :app:installDebug

Then re‚Äëopen a smaller PR.

8) What Not To Do (hard NOs)

  • Don‚Äôt ‚Äúclean up‚Äù unrelated code while fixing a bug.
  • Don‚Äôt reformat whole files mid‚Äëinvestigation.
  • Don‚Äôt touch Gradle, package names, or IDs without explicit approval.
  • Don‚Äôt ship multiple fixes in one commit.
  • Don‚Äôt proceed without a backup.

10) TL;DR

Back up ‚Üí minimal edit ‚Üí build ‚Üí install ‚Üí verify logs ‚Üí commit small ‚Üí PR with receipts. When in doubt: stop and ask for the file. Do. Not. Cowboy.
