Chief ‚Äî Worker Training

*

1) Communication & Operating Style

  • One step at a time. Run one command, paste the output, wait for confirmation.
  • Label every command with the correct window: [ANDROID] or [BACKEND]/[SERVER].
  • Be direct; no filler. Push back on weak ideas; protect stability over speed.
  • When confused, stop and verify. Truth over optimism.

2) Session Initialization Protocol (Every New GPT)

Run this to orient before acting. Keep it lightweight.

When to run:

  • At the start of a new chat/session.
  • Again only when you switch branches or change repos.
  • Do not re-run for tiny edits in the same branch/repo.
# [ANDROID] ‚Äî ~/AndroidStudioProjects/Chief
pwd && git status && git branch -vv && git log --oneline -n 5 && git describe --tags --always

# [BACKEND] ‚Äî ~/AndroidStudioProjects/chief-tts-backend
pwd && git status && git branch -vv && git log --oneline -n 5 && git describe --tags --always

# Global git & toolchain sanity (print-only)
git config --get core.autocrlf && git config --get core.eol && git --version
node -v || true; npm -v || true; java -version || true; ./gradlew -v || true

Confirm you understand: current branch, last known‚Äëgood tag, pending commits, which repos are active, and whether the worktree is clean. If the directive doesn‚Äôt match the branch‚Äôs purpose, stop and clarify.

3) Windows & Paths

  • [ANDROID] ‚Üí ~/AndroidStudioProjects/Chief
  • [BACKEND]/[SERVER] ‚Üí ~/AndroidStudioProjects/chief-tts-backend
  • Never mix contexts. Always prefix commands with the window label.

4) Git & EOL Policy (Windows‚Äësafe)

  • Keep repos clean before switching branches or building.
  • Prevent CRLF churn using a baseline .gitattributes (commit once):
* text=auto eol=lf
*.kt *.kts *.gradle *.sh *.c *.cpp *.h text eol=lf
  • Check for whitespace/CRLF issues before commits:
git diff --check || true

Pragmatic rule: don‚Äôt block harmless diffs, but prevent noisy churn and cross‚Äëplatform breakage.

5) Quick Commands (Copy/Paste)

Workspace Discovery (paste outputs):

pwd && git status && git branch -vv && git log --oneline -n 5 && git describe --tags --always

Open common files (Windows Git Bash ‚Üí Notepad):

notepad.exe "$HOME/AndroidStudioProjects/Chief/app/src/main/java/app/chief/app/MainActivity.kt"
notepad.exe "$HOME/AndroidStudioProjects/Chief/app/src/main/java/app/chief/app/ui/brief/BriefScreen.kt"
notepad.exe "$HOME/AndroidStudioProjects/Chief/app/src/main/java/app/chief/app/sync/SyncPoller.kt"

Backup (PowerShell via Windows Terminal):

$src = "$HOME/AndroidStudioProjects/Chief"
$dst = "E:\backups\Chief\$(Get-Date -Format yyyy-MM-dd_HH-mm)"
robocopy $src $dst /MIR /XD .git .gradle .idea build node_modules /R:1 /W:1

6) Branching Strategy (No‚ÄëChaos)

  • main ‚Üí stable, tagged, smoke‚Äëtested only.
  • feat/<version>-<scope> ‚Üí standard feature work.
  • spike/<topic> ‚Üí experimental sandbox work.
  • fix/<scope> ‚Üí hotfix branches (never commit fixes directly to main).

Always branch from the last known‚Äëgood tag.

git fetch --all --prune
# Replace <tag> and <branch>
git switch -c spike/<topic> <tag>

Examples:

spike/whisper-ndk-skel-safe  <- from restore/good-2025-11-09
feat/v4.0a.1-brief-swipe     <- from v4.0a.0

7) Commit Discipline

  • One commit = one logical change. No bundling.
  • Format: type(scope): summary
  • Types: feat, fix, chore, docs, refactor, test, build

Examples:

fix(brief-ui): restore horizontal swipe detection
chore(ndk): add JNI skeleton files (no wiring)
feat(tts): add onboarding greeting endpoint /onboarding/greet

8) Tagging & Checkpoints

  • Tag stable, working milestones (behavior verified) before touching a new subsystem.
  • For in‚Äëprogress breadcrumbs, you may use local lightweight tags or just clear commit messages. Do not push lightweight tags.
# Stable milestone (annotated, push)
git tag -a v4.0a.1 -m "brief swipe restored; JNI skeleton added"
git push --tags

# Breadcrumb (local only)
git tag temp/brief-swipe-WIP || true

Known‚Äëgood tags act as recovery anchors (e.g., restore/good-2025-11-09, v3.9-oauth-restored-2025-11-07).

9) Proof Artifacts & Naming

Require proof logs only for behavior‚Äëaffecting changes (build/UI/NDK/network/audio/auth). Skip logs for docs/comments or trivial refactors.

  • Store logs under proof/ with a timestamped name when required:
proof/<topic>-<YYYYMMDD-HHMM>.txt
  • Example: proof/brief-swipe-20251109-1950.txt
  • Include proof paths in handovers and PRs only when behavior changed.

10) Android Smoke Test

Run when behavior changes or before merging/release. Not needed for docs/comment‚Äëonly commits.

# [ANDROID]
./gradlew clean assembleDebug && ./gradlew installDebug
adb reverse tcp:8080 tcp:8080 || true
adb logcat -c
adb logcat -v time BriefScreen:I SyncPoller:I Replay:I *:S | tee proof/brief-swipe-$(date +%Y%m%d-%H%M).txt

Pass if: BUILD SUCCESSFUL, app launches, onboarding greeting plays, pulse + brief work, no crashes.

11) Backend Smoke Test

Run when backend behavior changes or before merging/release. Not needed for docs/comment‚Äëonly commits.

# [BACKEND]
node --version
npm --version || true
node server.cjs
# Expect: "Listening on http://localhost:8080"

If missing, investigate before claiming success.

12) Sandbox & Branch Protocol (No‚ÄëChaos)

  • Work from last known‚Äëgood tag; keep experiments quarantined to spike/‚Ä¶.
  • No ‚Äúcowboy merges.‚Äù Merge to main only after smoke tests and a tag.
  • Don‚Äôt copy files from unrelated branches without verification.
  • If the sandbox breaks, freeze, tag the last good commit, branch a new attempt from the good tag.

13) Guardrails (Prime Directives)

  • Protect stability. Speed comes second.
  • Never switch branches with a dirty worktree.
  • Never rewrite history above a known‚Äëgood tag.
  • If anything feels off: stop, tag, and re‚Äëbranch from the last good point.

14) Anti‚ÄëCowboy Protocol (Forbidden Moves)

  • Bundling unrelated changes into one commit.
  • Pushing broken code or skipping smoke tests.
  • Merging to main without an annotated tag + proof log.
  • Ignoring CRLF churn that causes noisy diffs across platforms.

15) Recovery & Safety Protocols

When things go sideways:

  1. Stop edits immediately.
  2. Tag the last working commit.
  3. Create a fresh branch from that tag:
git switch -c spike/<topic>-attempt2 <tag>
  1. Reapply minimal diffs, build, test, tag.

Rescue when local repo is tangled:

git stash -u
git fetch --all --prune
git switch -c rescue/$(date +%Y%m%d)-<topic> <last-known-good-tag>

Manually reapply only what‚Äôs necessary. Rebuild, test, tag.

Backups (Windows): run after tagging and testing; keep 5 most recent per project.

$src = "$HOME/AndroidStudioProjects/Chief"
$dst = "E:\backups\Chief\$(Get-Date -Format yyyy-MM-dd_HH-mm)"
robocopy $src $dst /MIR /XD .git .gradle .idea build node_modules /R:1 /W:1

16) Device Identity Protocol (Debugging)

  • Be explicit which device/emulator is under test.
  • If behavior differs (e.g., offline brief overlap vs server‚Äëconnected), capture two proof logs with device identifiers.
  • When swapping devices, clear app data or reinstall to avoid stale auth state.

17) Do‚ÄëNot‚ÄëTouch Without Tag (High‚ÄëRisk Files)

  • app/src/main/java/app/chief/app/MainActivity.kt
  • app/src/main/java/app/chief/app/ui/brief/BriefScreen.kt
  • app/src/main/java/app/chief/app/sync/SyncPoller.kt

If you must touch these, create a checkpoint tag first.

18) Handover Protocol (When Chat Nears Capacity)

Include only what the next worker needs, nothing extra:

Branch: <name> @ <short-sha>
Baseline Tag: <tag>
Proof: proof/<topic>-<YYYYMMDD-HHMM>.txt
Change Summary: one or two lines
Open Issues: file + line refs
Next Step: [WINDOW] <single command to run next>

One next step, not a bundle.

19) Definition of Done (Per Task)

  • Code compiles; smoke tests pass.
  • Commit is scoped and well‚Äënamed.
  • Annotated tag created and pushed.
  • Handover updated with proof.
  • Backup written.
  • Workspace clean.

Final Word: No chaos, no guessing, no dirty branches. Tag truth, build slow, stay clean.
